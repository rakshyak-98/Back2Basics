# x11

> One-line: **X Window System protocol and auth** — `$DISPLAY`, MIT-MAGIC-COOKIE-1, and SSH `-X`/`-Y` forwarding. Still relevant wherever XWayland or remote GUI appears.

## Mental model

X11 is a **network-transparent** protocol (though mostly local today). The **X server** (Xorg) owns the screen; **X clients** (apps) connect and request draws/input. Auth uses a **magic cookie** in `~/.Xauthority` — possession = ability to spy on or inject into your session.

```
App (X client) ──TCP/UNIX──► Xorg (X server) ──► GPU
SSH -X ──► forwards cookie + DISPLAY to remote app's connection back to your laptop's X server
```

| Concept | Location / tool |
|---------|-----------------|
| Display | `$DISPLAY` (`:0`, `hostname:10`) |
| Auth cookie | `~/.Xauthority` |
| `xauth` | List/add/remove cookies |
| `xhost` | Host-based ACL (legacy, coarse) |

**SSH forwarding:**

- `-X` — untrusted forward (restrictions on certain X extensions).
- `-Y` — trusted forward (more breaks, more risk).

## Standard config / commands

**Inspect current auth:**

```bash
echo $DISPLAY
xauth list
# hostname/unix:0  MIT-MAGIC-COOKIE-1  <hex secret>
```

**SSH with X forwarding:**

```bash
ssh -X user@remote          # untrusted
ssh -Y user@remote          # trusted — only if you trust remote admins/users
# Remote:
echo $DISPLAY               # localhost:10.0
xauth list
xeyes &                     # quick test
```

**Server sshd (must allow — default often no):**

```
# /etc/ssh/sshd_config
X11Forwarding yes
X11DisplayOffset 10
```

**Merge cookie manually (broken forwarding):**

```bash
# On laptop — extract cookie for :0
xauth list $HOSTNAME/unix:0
# On remote — add matching line
xauth add $HOSTNAME/unix:0 MIT-MAGIC-COOKIE-1 <hex>
export DISPLAY=...
```

**Allow local docker user (narrow):**

```bash
xauth nlist $DISPLAY | sed -e 's/^..../ffff/' | xauth -f ~/.Xauthority nmerge -
# Prefer: don't run untrusted containers with X access
```

## Triage (when things break)

| Symptom | Check | Fix |
|---------|-------|-----|
| `Can't open display` | `echo $DISPLAY` | Export `:0`; SSH with `-X`; on Wayland use XWayland |
| `No protocol specified` | `xauth list` vs display | Regenerate cookie; re-SSH |
| Forwarding disabled | `ssh -v` X11 debug | Enable `X11Forwarding` on sshd; `ForwardX11 yes` in ssh_config |
| App slow over SSH | Latency on X11 | Use native/web UI; `-C` compression hurts CPU |
| Blank xeyes | Firewall on X11 TCP | Prefer UNIX socket local; don't expose X11 TCP to network |
| Cookie leak concern | `ps e` remote users | Never `xhost +`; revoke with `xauth remove` |

## Gotchas

> [!WARNING]
> **SSH `-Y` trusted X11** — remote root/user can keylog and screenshot your desktop. Use `-X` or no GUI.

> [!WARNING]
> **Exposing X11 on TCP port 6000+n** — unencrypted remote control of screen. Disable `ListenTCP` in xorg.conf.

- **Wayland session** — apps may be Wayland-native; `$DISPLAY` set only for XWayland apps (`:0` or `:1`).
- **Cookie in `/tmp`** — some DM setups regenerate on login; long SSH sessions lose access until re-forward.
- **Docker `-e DISPLAY`** — mounts `~/.Xauthority` + `/tmp/.X11-unix` — only for trusted images.

## When NOT to use

- **New local Linux desktop** — prefer native Wayland apps; X11 is compatibility layer.
- **Production server administration** — SSH + CLI; no X forwarding on hardened hosts.
- **High-performance GUI over WAN** — use RDP/VNC/NoMachine, not X11 over SSH.
